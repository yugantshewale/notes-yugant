What is a “Token” in Django (or APIs)?
-> A token is like a temporary password/key that proves: “This user is logged in and allowed to access protected APIs.”
   Simple analogy
   You sign up / login → server gives you a token
   You use that token for future requests instead of sending password again
   Like:
   Login = getting a gate pass
   Token = showing that pass every time you enter

   ---------------------------------------------------------------------------------
   Why tokens are used
   Instead of sending:
   {
     "email": "test@test.com",
     "password": "12345678"
   }
   Sending this every time ❌ (unsafe)

   You send:
   Authorization: Token abc123xyz
   ✔ safer
   ✔ faster
   ✔ standard for APIs

   ---------------------------------------------------------------------------------
   How it works (flow)
   1. Signup/Login

   You create user → generate token:

   token, _ = Token.objects.get_or_create(user=user)
   2. API response
   {
     "token": "a94f8c2e1b7d..."
   }
   3. Client stores token
   Mobile app / frontend saves it
   4. Future requests
   Authorization: Token a94f8c2e1b7d...

   ---------------------------------------------------------------------------------
   ⚙️ In Django REST Framework (DRF)

   When you use:
   from rest_framework.authtoken.models import Token

   Django creates something like:
   {
     "user": 1,
     "token": "a94f8c2e1b7d..."
   }

python3-dev
	Contains Python header files and build tools

*** Serializers.py ***

In Django REST Framework (DRF), a serializer is a component that converts complex data (like Django models or querysets) into native Python 
data types, which can then be easily rendered into JSON, XML, etc.—and vice versa.
Serializers allow complex data such as querysets and model instances to be converted to native Python datatypes that can then be easily rendered 
into JSON, XML or other content types. 
Serializers also provide deserialization, allowing parsed data to be converted back into complex types, after first validating the incoming data.
The serializers in REST framework work very similarly to Django's Form and ModelForm classes.

In simple terms:
A serializer = translator between:
Django models / Python objects → JSON (API response)
JSON (client request) → Python objects → validated data → database

Serialization flow
JSON / XML (request data)
        ↓
Parser
        ↓
Python native datatypes
        ↓
Serializer (validation happens here)
        ↓
Validated data
        ↓
Saved as Django Model instance (database)


Deserialization 
Deserialization = converting incoming API data (JSON) into usable Python data + validating it + (optionally) saving it

Think of it like this
A client (frontend/Postman) sends:

{
  "name": "Alice",
  "age": 22
}
👉 This is JSON (text format) — Django can’t directly save this to DB

So Django REST Framework does:
JSON → Python dict → validate → create model object

Step-by-step

1. Request comes in (JSON)
{
  "name": "Alice",
  "age": 22
}

2. Parser converts JSON → Python dict
{"name": "Alice", "age": 22}

3. Serializer takes this data
serializer = StudentSerializer(data=request.data)

4. Validation happens
serializer.is_valid()
Checks:
required fields
field types (age must be int)
custom validation rules

5. Access validated data
serializer.validated_data
👉 Now it’s clean, trusted Python data

6. Save to database
serializer.save()

✔ Serializer does:
Validate data
Return cleaned Python data (validated_data)
Optionally transform/modify data (only if you write logic)
Create/update Django model objects (when .save() is called)
